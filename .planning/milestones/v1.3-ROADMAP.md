# Milestone v1.3: Split Match Scoring

**Status:** SHIPPED 2026-05-23
**Phases:** 13-15
**Total Plans:** 8

## Overview

v1.3 added a session-level split-match mode. Admins can now enable split scoring per session, finish matches as `2-0` or `1-1`, persist two game-level result rows per scheduled match, and keep all player-facing stats surfaces correct without breaking legacy one-game sessions.

## Phases

### Phase 13: Split Scoring Schema

**Goal**: Database and TypeScript types can represent session-level split scoring and multiple game results per scheduled match
**Depends on**: Phase 12
**Requirements**: FMT-01, RES-03, RES-04, COMP-01
**Plans**: 2 plans

Plans:
- [x] 13-01-PLAN.md - Split scoring schema migration, composite uniqueness, typed contract, and compatibility test baseline
- [x] 13-02-PLAN.md - App-side result compatibility helper plus reader/writer normalization for game-1 legacy behavior

**Details:**
- Added `sessions.split_match_scoring` and `match_results.game_number`
- Enforced duplicate-game protection with `(match_id, game_number)` uniqueness
- Preserved legacy one-game semantics through compatibility helpers and explicit `game_number: 1` writes

## Milestone Summary

**Key Decisions:**
- Session-level split scoring was chosen instead of per-match configuration
- `1-1` is a valid final result and each game win counts independently in stats
- Legacy one-game semantics remain authoritative when split scoring is disabled

**Issues Resolved:**
- Removed single-result assumptions from result readers and stats surfaces
- Closed split result-entry gaps with real browser/database validation
- Reconciled milestone docs, requirements, and validation coverage before closeout

**Issues Deferred:**
- No new product gaps were deferred from v1.3 closeout

**Technical Debt Incurred:**
- Planning history remains verbose in the main roadmap until a later archival cleanup
- Supabase migrations still rely on Dashboard SQL execution on Windows
