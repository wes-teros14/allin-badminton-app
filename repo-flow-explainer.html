1. Overview

A club-management web app for a recurring badminton session: an admin sets up a session, players register, the admin generates a fair doubles match schedule (an in-browser optimizer, no server-side job), the group plays through the queue court-by-court with live cross-device sync, and results feed an all-time leaderboard. There is no custom backend process — Supabase (Postgres + Row Level Security + Realtime + a handful of SQL trigger functions) is the entire server side; the React app talks to it directly via @supabase/supabase-js.

React 19.2 TypeScript Vite 8 (rolldown) react-router 7.13 @supabase/supabase-js 2.99 Tailwind CSS 4.2 Zod 4 + react-hook-form 7 Vitest 2 (unit) Playwright 1.58 (e2e) Supabase CLI (migrations)

Versions are the caret ranges from package.json (repo root of the app is badminton-v2/, the rest of this document's paths are relative to that folder).

3. Process flow walkthrough

Session lifecycle: setup → registration → schedule → live play → complete

sessions.status is the state machine (defined in SessionStatus, src/hooks/useSession.ts:7-13): a strict enum of six values. Every transition is a plain, unguarded (no DB constraint enforcing legal transitions) function in useSession() (src/hooks/useSession.ts:64-366), called from button handlers in SessionView (src/views/SessionView.tsx:347-566), which renders a completely different panel per status value (src/views/SessionView.tsx:487-563).

  1. setup. SetupCard (src/views/SessionView.tsx:172-345) collects name/date/venue/price/court count. Clicking confirm calls openRegistration() (src/hooks/useSession.ts:139-169), which inserts a session_invitations row (the shareable registration link's token doubles as that row's primary key) then flips sessions.status to registration_open.
  2. registration_open. RegistrationURLCard shows the shareable link and lets the admin set a player cap and court count; players self-register via src/views/RegisterView.tsx (skimmed). RosterPanel (non-editable mode, src/views/SessionView.tsx:511) lists who has joined. closeRegistration() (src/hooks/useSession.ts:207-236) deactivates the invitation and moves to registration_closed. Re-opening later (reopenRegistration(), src/hooks/useSession.ts:171-205) reactivates the same invitation row rather than creating a new one — see the gotchas section for why that distinction was a real bug once.
  3. registration_closed — roster overrides + schedule generation (branch point, most non-obvious stage). Two sibling panels render together here: RosterPanel (editable, src/views/SessionView.tsx:524) and MatchGeneratorPanel (src/views/SessionView.tsx:525). Each owns its own independent copy of the player list via two separate hooks — useRoster() (src/hooks/useRoster.ts:39-149) and useRegisteredPlayers() (src/hooks/useRegisteredPlayers.ts:45-125) — there is no shared client-side cache anywhere in this app (no React Query, no Zustand, no Context store for this data; confirmed absent from package.json). Per-session gender/level overrides are edited here (updateSessionOverride, src/hooks/useRoster.ts:130-138, writes to session_registrations.gender/level — never touches profiles). This used to only reach the match generator after a full page refresh (both hooks' Realtime channels subscribed to a table that was never added to the Realtime publication); SessionView.tsx:362 now holds a rosterVersion counter, bumped by RosterPanel's onRosterChange callback and threaded into useRegisteredPlayers(sessionId, rosterVersion) (src/components/MatchGeneratorPanel.tsx:93) so a roster edit now forces an immediate refetch. See the gotchas section for the full story and why this approach was chosen over fixing Realtime. Clicking Generate calls handleGenerate() (src/components/MatchGeneratorPanel.tsx:211-287), which validates every player has a level (and gender, unless gender rules are disabled) then calls generateScheduleOptimized() (src/lib/matchGenerator.ts:745-820, called at src/components/MatchGeneratorPanel.tsx:269) — a pure, synchronous, in-browser simulated-annealing optimizer (see the function reference for how it works and the empty-schedule bug it used to have). The result previews in the same panel; Save & Lock calls lockSchedule() (src/hooks/useSession.ts:238-272), which bulk-inserts every match row (queue_position = array index + 1) and moves to schedule_locked. unlockSchedule() (src/hooks/useSession.ts:274-293) is the reverse: deletes every match row for the session and returns to registration_closed.
  4. schedule_locked. MatchGeneratorPanel, re-rendered read-mostly (loads the locked matches via src/components/MatchGeneratorPanel.tsx:109-144, keyed on sessionStatus === 'schedule_locked'), lets the admin swap two players in a still-queued match (handleEditSave, src/components/MatchGeneratorPanel.tsx:294-324) before play starts. startSession() (src/hooks/useSession.ts:295-334) takes the first N queued matches (N = configured court count, via normalizeCourtCount, src/lib/courts.ts:13-16), assigns each a court number via buildStartingCourtAssignments (src/lib/courts.ts:73-78) and a started_at timestamp, flips each to playing, then moves the session to in_progress.
  5. in_progress — live play. LiveSessionView (src/views/SessionView.tsx:126-170) replaces the whole page with CourtTabs (src/components/CourtTabs.tsx:305-...), fed by useAdminSession() (src/hooks/useAdminSession.ts:52-223) and kept live by useRealtime() (src/hooks/useRealtime.ts:11-91) — this is the genuinely asynchronous part of the app and gets its own walkthrough below. Finishing a match and auto-promoting the next queued one onto the freed court both happen inside one function, markDone() (src/hooks/useAdminActions.ts:72-125). unstartSession() (src/hooks/useSession.ts:336-351) is the admin's undo: reverts every playing/queued match back to queued with cleared court numbers, back to schedule_locked.
  6. complete. closeSession() (src/hooks/useSession.ts:353-363) sets status = 'complete' and completed_at = now(). This single UPDATE is also the trigger for the sessions-attended count update described in the next flow — nothing else in the client fires as a result of reaching this state; SessionView.tsx:413-427 then just renders a static "This session has been closed" card.

Live sync & stats aggregation (server-driven, not client-triggered)

Two separate mechanisms, both entirely inside Supabase, run independently of which client made the original write:

(a) Cross-device Realtime sync during live play.

  1. Any admin device (or the read-only LiveBoardView / TodayView spectator pages, skimmed) opens useRealtime(sessionId, onUpdate, channelPrefix) (src/hooks/useRealtime.ts:11-91), which subscribes to Postgres change events on three tables: matches filtered by session_id, all of match_results, and sessions filtered by id (src/hooks/useRealtime.ts:28-64).
  2. Any write to those tables from any connected client — another admin's device finishing a match, a substitute being swapped in — broadcasts a postgres_changes event to every subscriber, which calls the passed-in onUpdate callback (here, refresh from useAdminSession) to refetch.
  3. The hook also tracks its own subscription health as connected / reconnecting / disconnected (src/hooks/useRealtime.ts:65-83), surfaced to the admin via LiveIndicator (src/components/LiveIndicator.tsx, skimmed) so a dropped WebSocket connection is visible rather than silently stale.

(b) DB-trigger-driven stats aggregation. Entirely SQL, invisible from any client code — this is why the client never writes to player_stats or player_pair_stats directly (both are read-only to clients, per their RLS grants).

  1. markDone() inserts one row into match_results (src/hooks/useAdminActions.ts:96-100). That single INSERT, from whichever admin device happened to click Done, fires update_player_stats_on_result() (supabase/migrations/013_player_stats_tables.sql:68-131, trigger on_match_result_insert at line 134), which looks up the four players on that match, increments games_played/wins for all four in player_stats, and increments wins_together / losses_against for every pairing in player_pair_stats — one write becomes up to 4 + 2 + 4 = 10 upserts, all inside the same trigger transaction.
  2. Separately, closeSession()'s UPDATE to sessions.status = 'complete' fires increment_sessions_attended_on_complete() (supabase/migrations/030_sessions_attended_on_completion.sql:12-31, trigger on_session_complete at line 34), which increments player_stats.sessions_attended for every player who was registered for that session. This trigger replaced an earlier version that incremented on registration instead of on completion (see the gotchas section for the backfill bug this transition caused).
  3. Both tables feed fetchAllTimeLeaderboard() (src/views/LeaderboardView.tsx:59-96), which is a pure client-side read against the already-aggregated numbers — no computation happens in the browser beyond the eligibility filter and sort (see the function reference).

No queues, cron jobs, or webhooks exist in the current app. An email-notification system built on pg_cron + pg_net (migrations 045-049: payment-confirmed, session-full, and schedule-ready triggers, plus two scheduled reminder cron jobs) was built and then fully reverted in supabase/migrations/050_rollback_email_system.sql — it does not exist in the current schema; unverified why it was rolled back (no comment in that migration beyond "Rollback: remove all email notification infrastructure").

4. Data model

Postgres via Supabase, RLS on every table. No ORM — every query in the client is a @supabase/supabase-js query-builder call against these tables directly; src/types/database.ts hand-maintains the TypeScript Database type (not generated in this repo's scripts). 72 migration files exist (supabase/migrations/001 through 072); the table below reflects each table's current shape as pieced together from the migrations that created and later altered it, not a fresh supabase db pull.

Player-facing routes are wrapped in PlayerLayout (top nav bar); admin-only routes (/admin, /session/:sessionId, /players, /inventory, /finance) are additionally wrapped in AdminRoute (src/App.tsx:10-30), which redirects unauthenticated users to a Google sign-in button and non-admin/moderator users to /.

Called by

Calls

Why it exists
Lazy-loading every view keeps the initial bundle to just the shell + whichever single view is actually needed; the admin/player split at the route level (rather than inside each view) means an unauthorized visit never even downloads the admin view's code.
useSession() -- registration transitions src/hooks/useSession.ts:139-236
What it does
openRegistration (139-169) inserts a fresh session_invitations row then sets status. reopenRegistration (171-205) instead looks up the most recent existing invitation row and flips is_active back to true — deliberately not symmetric with openRegistration, see the gotchas entry on why. closeRegistration (207-236) deactivates the invitation and sets status to registration_closed.
Inputs / outputs
No arguments (read session/invitation from the hook's own state); all three are void-returning, updating hook state as a side effect, with toast.error on any Supabase error.
Called by
src/views/SessionView.tsx button handlers (onConfirm at line 499, handleCloseRegistration, the Reopen Registration button at line 539)
Calls
supabase.from('session_invitations'/'sessions') directly — no further app-code calls
Why it exists
Keeping the registration link's identity (its id, which is the shareable URL token) stable across a close/reopen cycle matters because the link may already be shared; minting a new one on reopen would silently break every copy of the old link still in circulation.
useRoster() + useRegisteredPlayers() -- roster sync src/hooks/useRoster.ts:39-149, src/hooks/useRegisteredPlayers.ts:45-125
What it does
Two independent data-fetching hooks over the same underlying table (session_registrations joined to profiles), shaped differently for their two consumers: useRoster returns admin-editing-shaped RosterPlayer[] (with registrationId, paid) plus mutators addPlayer/removePlayer/updateSessionOverride/updatePaid (114-147); useRegisteredPlayers returns match-engine-shaped PlayerInput[] via the pure helper buildRegisteredPlayers (src/hooks/useRegisteredPlayers.ts:27-43). Both also open a Realtime channel on session_registrations, but see the gotchas section — those channels never actually deliver events.
Inputs / outputs
useRoster(sessionId, onChange?) — the optional onChange callback (added as this session's fix) fires after every successful addPlayer/removePlayer/updateSessionOverride. useRegisteredPlayers(sessionId, refreshKey?) — changing refreshKey forces its fetch useEffect to re-run (src/hooks/useRegisteredPlayers.ts:122, dependency array [sessionId, refreshKey]).
Called by
src/components/RosterPanel.tsx:27-29 and src/components/MatchGeneratorPanel.tsx:93 respectively, both rendered as siblings by src/views/SessionView.tsx:524-525, which owns the rosterVersion counter (line 362) that connects onChange on one side to refreshKey on the other.
Why it exists
This app has no shared client-side data cache of any kind, so any two components reading the same table independently will drift unless something explicitly re-synchronizes them. This onChange → rosterVersion → refreshKey wiring is that explicit resync, chosen over fixing the dead Realtime channel because it works regardless of whether Realtime's publication config is ever correct, and this exact class of bug (a table missing from the Realtime publication) had already silently broken a different feature once before (supabase/migrations/041_notifications_realtime.sql's comment).
generateScheduleOptimized() -- match engine entry src/lib/matchGenerator.ts:745-820
What it does
Multi-start simulated annealing: for numStarts (default 15, UI default 50) restarts, builds a fresh initial assignment via buildAssignment (784-787, with allowRelaxSpread=false) then improves it for numTrials (UI default 2000) iterations via optimizeAssignment (790-794), keeping whichever start scored highest across all restarts. Fully synchronous, runs entirely in the browser's main thread — src/components/MatchGeneratorPanel.tsx:238 wraps the call in a setTimeout(..., 50) purely so React can paint the loading spinner first.
Inputs / outputs
In: PlayerInput[] plus OptimizeOptions (match count, max spread, gender rules, scoring weights, wishlist pairs, rest-spacing settings). Out: { matches: GeneratedMatch[], audit: AuditData, decisions: MatchDecision[] } — audit drives every metric shown in the UI (Optimizer Score, Skill Gap Violations, etc., src/components/MatchGeneratorPanel.tsx:528-541).
Called by
handleGenerate(), src/components/MatchGeneratorPanel.tsx:211-287 (call site line 269)
Calls
buildAssignment (matchGenerator.ts:180), optimizeAssignment (matchGenerator.ts:437), assignmentToMatches (matchGenerator.ts:363), evaluateSessionScore (matchGenerator.ts:498)
Why it exists
Doubles scheduling has competing goals (even game counts, skill-balanced teams, gender-composition rules, minimal repeat partnerships, rest spacing) that don't have one obvious greedy solution; simulated annealing lets one weighted scoring function (evaluateSessionScore) trade those goals off against each other and searches many random restarts for a good combination instead of hand-coding a scheduling algorithm.
buildAssignment() + findValidGroup() / pickGroup() src/lib/matchGenerator.ts:180-239, 245-271, 277-304
What it does
Greedily builds one full round of matches: each iteration picks the 4 candidates with the most remaining games owed (ties broken by who didn't just play, then a pre-shuffle), and calls findValidGroup (245-271) to pick a spread/gender-valid foursome from them via progressive relaxation — try strict spread+gender, then spread-only, and (only if allowRelaxSpread is true) relax spread too. pickGroup (277-304) does the actual exhaustive search over the top-16 candidates by need.
Inputs / outputs
In: candidate player IDs, level/gender maps, match count, spread limit, gender-rules flag, allowRelaxSpread. Out: string[][], one array of 4 player IDs per match — as of this session's fix, always exactly numMatches long (see below).
Called by
generateSchedule (matchGenerator.ts:674, single-pass API, allowRelaxSpread defaults true) and generateScheduleOptimized (matchGenerator.ts:784, SA's Phase 1 seed, explicitly passes false)
Why it exists / recent fix
Bug fixed this session: when allowRelaxSpread=false (the SA seed call) and no 4-player group anywhere satisfied the spread limit, the old fallback (candidates.slice(0, 4)) was gated behind allowRelaxSpread too — so that match slot was silently skipped entirely. With a roster whose levels made every possible foursome exceed the UI's default "Max Skill Gap" of 2, every slot got skipped, buildAssignment returned [], and the whole generator produced a 0-score, 0-match, no-error result — indistinguishable from the button doing nothing. The fix (src/lib/matchGenerator.ts:229) removed the allowRelaxSpread && guard from that fallback, so a slot is always filled with the best-available four candidates; the SA phase and the audit then surface the real constraint violation ("Skill Gap Violations: N") instead of the generator going silent.
optimizeAssignment() -- simulated annealing src/lib/matchGenerator.ts:437-493
What it does
Standard SA loop: starting temperature 500 cooling geometrically to 1 over numTrials steps (462-465). Each step mutates the current assignment — 80% of the time a mutateCrossSwap (391-420, swaps one player between two matches, rejected if either resulting group would exceed the spread limit), else a mutateRowSwap (425-432, just reorders two matches, for streak/rest optimization) — and accepts the move if it scores better, or with declining probability (Math.exp(delta / temperature)) if it scores worse. Tracks the best-ever assignment seen across the whole run.
Inputs / outputs
In: an initial assignment plus every scoring/constraint parameter. Out: { assignment, audit } for the best assignment found.
Called by
generateScheduleOptimized (matchGenerator.ts:790), once per start
Calls
assignmentToMatches + evaluateSessionScore (via the local scoreAssignment closure, 451-455), on every trial
Why it exists
buildAssignment's greedy pass gets participation counts right by construction but has no sense of overall schedule quality (streaks, repeat partners, rest spacing); SA is the mechanism that improves those without breaking the participation-count guarantee the greedy pass already established.
markDone() src/hooks/useAdminActions.ts:72-125
What it does
The single function behind "finish this match": marks the match complete (guarded by .eq('status', 'playing') so a double-click or two admins clicking at once can't double-process it — the second call's update matches zero rows and short-circuits at line 90), records the result (submitSplitResult for split scoring, or a plain match_results insert otherwise), then looks up the next queued match by queue_position and promotes it onto the just-freed court.
Inputs / outputs
In: matchId, courtNumber, sessionId, optional winningPairIndex / startedAt / splitOutcome. Side effects only (DB writes, toast.error on failure, onDone() callback on success); no return value.
Called by
src/components/CourtTabs.tsx (via CourtCard's onMarkDone prop, wired at src/components/CourtTabs.tsx:414)
Calls
supabase.from('matches'/'match_results') directly; indirectly fires the update_player_stats_on_result SQL trigger (see the async flow above)
Why it exists
Combining "finish" and "promote next" into one function means a freed court is never left idle waiting for a second admin action — the queue drains itself automatically as long as there's anything left in it.
useRealtime() src/hooks/useRealtime.ts:11-91
What it does
Opens one Supabase Realtime channel per (sessionId, channelPrefix) pair, subscribed to postgres_changes on matches (filtered by session), all of match_results (unfiltered — every session's results broadcast to every open channel, filtered client-side by whatever onUpdate chooses to do), and sessions (filtered by id). Also tracks connection status.
Inputs / outputs
In: sessionId, an onUpdate callback, an optional channel-name prefix (defaults 'live-board', but LiveSessionView passes 'session' so its channel name doesn't collide with a same-session LiveBoard tab open in another window). Out: { status, refresh }.
Called by
src/views/SessionView.tsx:129 (admin live view), LiveBoardView.tsx and TodayView.tsx (both skimmed, spectator views using the default 'live-board' prefix)
Calls
supabase.channel(...).on('postgres_changes', ...).subscribe(...) directly — this table set (matches, match_results, sessions) is correctly registered in the Realtime publication (supabase/migrations/010_realtime_replica_identity.sql, 064_add_sessions_and_results_to_realtime.sql) — contrast with session_registrations, which is not (see gotchas).
Why it exists
Live play is the one part of this app where multiple people (an admin on a laptop, a spectator on the LiveBoard TV display, players checking /today on their phones) are expected to be looking at the same session simultaneously; this is the only place actual push-based sync is load-bearing rather than a nice-to-have.
update_player_stats_on_result() (SQL) supabase/migrations/013_player_stats_tables.sql:68-131
What it does
AFTER INSERT ON match_results trigger. Looks up the 4 players on the referenced match, determines winners/losers from winning_pair_index, then upserts player_stats.games_played/wins for all 4 and player_pair_stats.wins_together/losses_against for every winner-loser and winner-winner pairing (both directions stored separately, so this is a directed graph, not an undirected one).
Inputs / outputs
Standard Postgres trigger signature (NEW row = the inserted match_results row). No return value visible to callers; SECURITY DEFINER so it can write tables the inserting client role can't write directly (client grants on player_stats/player_pair_stats are SELECT-only).
Called by
Postgres itself, on every match_results insert — currently only markDone and submitSplitResult insert there
Why it exists
Doing this aggregation as a trigger rather than client-side means it happens exactly once per result no matter which client (or how many concurrent admin devices) recorded it, and the leaderboard never depends on the client staying online long enough to finish a multi-step aggregation.
fetchAllTimeLeaderboard() src/views/LeaderboardView.tsx:59-96
What it does
Reads player_stats filtered to games_played > 0 AND sessions_attended >= 3, separately reads the 4 most recently complete sessions (RECENT_SESSIONS_WINDOW = 4, line 57) and everyone registered in any of them, then keeps only players who satisfy both the lifetime-sessions gate and the recent-activity gate, sorted by win rate (ties broken by raw win count), top 10.
Inputs / outputs
No arguments. Out: LeaderboardEntry[] (playerId, displayName, avatarUrl, wins, losses, winRate).
Called by
WinsLeaderboard (src/views/LeaderboardView.tsx:122-162), the "Mga Lodi" tab of LeaderboardView
Calls
Three parallel Supabase queries (player_stats, profiles, sessions), then one more sequential query against session_registrations once the recent session IDs are known
Why it exists
Two real tuning problems, each fixed with its own gate: ranking by raw win rate alone let a player with a tiny sample (e.g. 1 win, 0 losses) outrank regulars, hence the sessions_attended >= 3 minimum; and a player with a great historical win rate who stopped attending would otherwise sit at the top indefinitely, hence the separate "registered in one of the last 4 completed sessions" recency gate layered on top. The caption string on line 145 ("Ranked by win rate · min. 3 sessions played · must be active in the last 4") is this function's two filters spelled out for the reader.

8. Glossary

SessionOne scheduled club meetup, the top-level entity everything else hangs off of. Progresses through 6 statuses (SessionStatus, src/hooks/useSession.ts:7-13).
Roster overrideA session-scoped gender/level value on session_registrations that takes precedence over a player's default profiles.gender/profiles.level for that one session only — used e.g. when someone's actual skill that day differs from their usual rating.
Queue positionmatches.queue_position — the fixed play order within a session, assigned once at schedule-lock time and never renumbered afterward (courts are assigned dynamically from whichever match is next-queued, not by editing this field).
Court numbermatches.court_number — which physical court a playing match is on; NULL for queued/complete matches. Distinct from queue_position.
Skill gap / spreadThe max-minus-min player level within one 4-player match (getSpread, src/lib/matchGenerator.ts:164-167); "Max Skill Gap per Match" in the UI is the hard limit on this value.
Simulated annealing (SA)The optimization technique in optimizeAssignment: accept better moves always, worse moves with a probability that shrinks as a "temperature" parameter cools over the run — lets the search escape local optima early and converge late.
Assignment (matrix)The intermediate string[][] representation used during generation — one array of exactly 4 player IDs per match, before team-splitting (formTeams) or level/type computation happens.
Realtime publicationA Postgres/Supabase configuration list of which tables broadcast postgres_changes events at all; a table missing from it means any client subscription to it silently never fires, with no error anywhere.
rosterVersion / refreshKeyThis session's fix: a plain incrementing counter passed as a React prop specifically to force a dependent useEffect to re-run, used here to resync two sibling components that don't share a data cache.
Mga LodiTagalog slang roughly meaning "the greats/legends" — the label on the win-rate All-time Leaderboard tab (src/views/LeaderboardView.tsx), as opposed to the separate Cheers and Awards tabs.
Split scoringA per-session toggle (sessions.split_match_scoring) whose downstream result-recording path (submitSplitResult) was not traced in this pass — noted here, not explained.
