# Product Requirements Document - badminton_v2

**Author:** Wes
**Date:** 2026-03-18

## Executive Summary

badminton_v2 is a React + Supabase SPA that manages the complete lifecycle of a badminton group session — player registration, match scheduling, live court display, and cumulative player statistics — rebuilt to replace a Streamlit app that hit its ceiling on UI flexibility, view separation, and real-time capability. The system serves 10–30 users across three roles (Admin, Player, Kiosk) with role-based authentication, live push updates, and zero operational cost on free-tier infrastructure.

The core problem v1 could not solve: Streamlit's component model made distinct, purpose-built views impractical — admin, player, and kiosk experiences were crammed into the same surface with no real-time updates (manual polling only). v2 delivers a proper SPA with role-separated views that update live via Supabase Realtime.

As a secondary improvement, v2 eliminates the Google Forms → Google Sheets → manual copy-paste registration pipeline. Players register directly in the app via Google OAuth sign-in using a time-limited unique URL generated by admin — no external tools required.

### What Makes This Special

Three purpose-built views (Admin, Player, Kiosk) sharing live data via Supabase Realtime — no page refreshes, no polling, no external tools. Registration, scheduling, live court display, and player stats are all self-contained in one app.

## Project Classification

- **Project Type:** Web App (SPA)
- **Domain:** General / Sports-Recreation
- **Complexity:** Low
- **Project Context:** Brownfield rebuild — same core capabilities as v1, new stack and architecture
- **Stack:** React (frontend) + Supabase (PostgreSQL + Realtime + Auth with Google OAuth)
- **Users:** 10–30 members, free-tier only, no operational cost

## Success Criteria

### User Success

- Players and admin see match schedule updates without manual refresh
- Kiosk display reflects schedule changes within ≤2 seconds of any admin or player action
- Any user reaches their relevant view without friction or confusion

### Business Success

- v2 fully replaces the Streamlit app with no reason to return
- Admin runs a complete session (registration → match generation → live display) without leaving the app or using any external tool

### Technical Success

- App remains available for the full duration of a session (typically 2–4 hours)
- All functionality operates within Supabase and hosting free-tier limits for 10–30 concurrent users
- Real-time updates delivered via Supabase push — no polling

### Measurable Outcomes

- Kiosk updates in ≤2 seconds after any admin or player action
- Match generation for 10–30 players completes within 5 seconds
- Zero manual data transfer steps required for admin during a session

## Product Scope

### Phase 1 — MVP

Minimum needed to retire the old Streamlit app:

- Match generator (create and manage full session schedule)
- Player view (public, no login — game number, matchups, queue status)

### Phase 2 — Full App Target (Primary Goal)

Everything needed to run sessions end-to-end:

- Kiosk view (two-panel landscape tablet, per-court queue + Finish button)
- Admin view (desktop for pre-session setup, mobile for day-of)
- In-app registration via Google OAuth + unique time-limited URL
- Role-based authentication (Admin / Player via Google OAuth / unauthenticated Kiosk)
- Bidirectional real-time sync (Supabase push, manual refresh as fallback)
- Cumulative player win/loss tracking and attendance history
- Session statistics and charts

### Phase 3 — Vision

- Player notifications ("your game is next")
- Historical session archive

### Risk Mitigation

- **Real-time:** Manual refresh accepted as fallback if Supabase Realtime proves unreliable on free tier
- **Resource:** Solo build — MVP ships incrementally; Phase 2 is the real target, built feature by feature
- **Infrastructure:** All components must remain within free-tier limits for 10–30 concurrent users

## User Journeys

### Journey 1: Admin — Pre-Session Setup (Days Before)

**Wes prepares for Sunday's session a few days in advance.** He opens the app on his laptop, logs in as admin, and opens the registration window — the app generates a unique random URL. He shares it with the group (e.g. via WhatsApp). Players click the link; those already signed in with Google land directly on the registration page, others are prompted to sign in first. When registration closes (admin manually closes it), Wes sees the confirmed player list and runs the match generator. The app produces a full schedule across two courts — all rounds, all players, sequenced. He reviews and saves it. The schedule is locked and ready.

**Capabilities revealed:** Admin auth, unique registration URL generation, registration open/close, Google OAuth sign-in, player self-registration, match generation, schedule review and save.

---

### Journey 2: Admin — Day-of Intervention

**During the session, Court 1's queue needs a manual fix.** Wes opens the app on his phone, sees the current match state across both courts, and edits the affected match. In v1, someone would have had to physically refresh the kiosk tablet. In v2, the kiosk picks up the change within 2 seconds automatically.

**Key v1 → v2 fix:** Admin → Kiosk push was broken in v1. v2 fixes bidirectional sync via Supabase Realtime.

**Capabilities revealed:** Admin remote match editing, bidirectional real-time sync (admin ↔ kiosk), court-level match state management.

---

### Journey 3: Players — Using the Kiosk

**Players arrive at the venue.** The tablet shows the kiosk view: two panels side by side — Court 1 and Court 2. Each panel shows the current game and the next queued game. When a game finishes, the players on that court hit "Finish" on their panel. The current game clears, the next slides up, and the one after appears as "up next."

**Capabilities revealed:** Two-panel kiosk layout, current + next game display per court, player-triggered Finish action, real-time queue advancement.

---

### Journey 4: Player — Personal View

**Marco checks his schedule on his phone.** His player view shows all his games for the session — game number, partner, opponents — in order, with completed games marked. Court assignment is not shown (courts are dynamic, determined by whichever finishes first). To know his court, Marco glances at the kiosk.

- Player view answers: *"What game am I, and who am I playing?"*
- Kiosk answers: *"What game number is on which court right now?"*

**Capabilities revealed:** Public player view (no login), game number + matchup display, real-time queue status, game number as the shared reference between views.

---

### Journey 5: Player & Admin — Stats and Attendance

**After a session wraps, players and admin check cumulative records.** Marco opens the app on his phone and navigates to his stats page — he can see his all-time win rate and a list of every session he attended. He shares his win rate with the group. Wes, as admin, opens the admin view to see stats across all players — useful for seeding future match generation or settling friendly debates about who's been winning most.

Attendance tracks automatically from the moment a player first registers; no manual entry required. Win/loss is recorded per game as the kiosk Finish button is pressed — the system knows the four players and which pair won based on the outcome entered.

**Capabilities revealed:** Player stats view (own win rate + attendance history), admin stats view (all-player win rates + cumulative attendance), automatic attendance recording, automatic win/loss recording per game.

## Web App Requirements

### Technical Stack

- **Rendering:** SPA (React) — single load, client-side routing between views
- **Real-time:** Supabase Realtime — bidirectional push (admin ↔ kiosk, kiosk ↔ player)
- **Auth:** Supabase Auth with Google OAuth provider
- **Data:** Supabase PostgreSQL

### Browser & Device Matrix

| View | Primary Device | Browser Target |
|---|---|---|
| Player view | Mobile phone | Modern mobile (Chrome, Safari) |
| Admin — pre-session setup | Desktop | Modern desktop (Chrome, Firefox, Edge, Safari) |
| Admin — day-of session | Mobile phone | Modern mobile (Chrome, Safari) |
| Kiosk view | Tablet (landscape) | Modern mobile (Chrome, Safari) |

### Responsive Design

- Mobile-first overall; Admin pre-session setup optimised for desktop (full match schedule visibility)
- Admin day-of must be fully functional on mobile
- Kiosk two-panel layout designed for landscape tablet orientation

### Constraints

- No SEO required — private tool, no public indexing
- No WCAG accessibility compliance required

## Functional Requirements

### Authentication & Identity

- FR1: Players can sign in using their Google account
- FR2: Admin can sign in using their Google account with elevated privileges
- FR3: Kiosk view is accessible without authentication
- FR4: Player view is accessible without authentication (public read-only)
- FR5: Admin can generate a unique registration URL for a session
- FR6: Admin can close registration, invalidating the current registration URL

### Player Registration

- FR7: A signed-in player can register for a session via the active registration URL
- FR8: A player who is not signed in is prompted to sign in before registering
- FR9: Admin can view the list of registered players for a session
- FR10: Admin can manually add or remove a player from a session's roster

### Match Generation & Schedule Management

- FR11: Admin can generate a full match schedule from the confirmed player roster
- FR12: Admin can review the generated match schedule before saving it
- FR13: Admin can save and lock the match schedule
- FR14: Admin can manually edit a match in the schedule after it has been generated
- FR15: The system maintains a per-court ordered queue of matches

### Session Execution — Kiosk

- FR16: The kiosk displays two courts side by side, each showing the current and next queued game
- FR17: Players can mark a game as finished on the kiosk, advancing that court's queue
- FR18: The kiosk updates in real time (within ≤2 seconds) when the match queue changes
- FR19: Admin can manually advance or adjust a court's queue remotely

### Session Execution — Player View

- FR20: Players can view all their matches for the session (game number, partner, opponents)
- FR21: The player view reflects completed games and remaining queue in real time (within ≤2 seconds)
- FR22: Player view does not display court assignment (court is dynamic, shown only on kiosk)

### Real-Time Sync

- FR23: Admin changes are pushed to the kiosk without manual refresh
- FR24: Kiosk Finish actions are reflected in the admin view in real time (within ≤2 seconds)
- FR25: A manual refresh fallback is available on the kiosk if real-time sync fails

### Admin Session Management

- FR26: Admin can create a new session
- FR27: Admin can set the number of active courts for a session
- FR28: Admin can view the current state of all courts and the full match queue during a session

### Player Statistics & Attendance

- FR29: The system records the win/loss outcome of each doubles game per player
- FR30: The system tracks cumulative win/loss record per player across all sessions
- FR31: Players can view their own all-time win rate
- FR32: Admin can view all-time win rate for all players
- FR33: The system records each player's attendance per session from first registration onward
- FR34: Admin can view cumulative attendance per player across all sessions
- FR35: Players can view their own attendance history across all sessions

## Non-Functional Requirements

### Performance

- Kiosk updates reflect within ≤2 seconds of any admin or player action
- All user-facing interactions complete within 500ms for 95th percentile actions (no spinner that makes users question if the action registered)
- Match generation for 10–30 players completes within 5 seconds

### Security

- All user data stored in Supabase, protected by row-level security
- Unauthenticated access limited to public Player view and Kiosk view only
- Registration URLs expire on close — old URLs must not grant access
- Admin privileges enforced server-side

### Reliability

- App remains available and functional for the full duration of a session (2–4 hours)
- Real-time sync failure must not crash or freeze the kiosk — manual refresh fallback keeps the session running
- Match schedule, results, and attendance data must not be lost on device disconnect/reconnect

### Scalability

- Supports 10–30 concurrent users within Supabase free-tier limits
- No growth planning required beyond this range — free-tier ceiling is an accepted constraint
